## Context

当前项目中的 AsyncQueue 抽象类已实现但未投入使用。现有基础设施存在以下问题：

1. **EventBus**：`emit()` 方法直接创建 `asyncio.create_task()`，高并发时可能创建数千个未限制的任务，导致内存压力
2. **Agent 执行**：缺乏统一的任务队列，无法限制并发执行数，可能耗尽系统资源
3. **WebSocket**：消息直接发送，没有缓冲机制，突发流量可能导致消息丢失

AsyncQueue 提供了理想的抽象来解决这些问题：背压控制、类型安全、统一的接口语义。

## Goals / Non-Goals

**Goals:**
- 为 EventBus 添加基于队列的背压控制，防止任务爆炸
- 为 Agent 执行提供任务队列，限制并发数（默认 5 个并发）
- 为 WebSocket 添加消息缓冲，平滑流量突发
- 所有实现复用已有的 AsyncQueue 抽象，不重复造轮子
- 提供配置选项，允许根据环境调整队列大小和并发数

**Non-Goals:**
- 不替换现有 EventBus 的 API（保持向后兼容）
- 不实现持久化队列（当前使用内存队列）
- 不改变 HTTP API 或对外接口
- 不引入新的外部依赖

## Decisions

### 1. QueuedEventBus 采用装饰器模式

**决策**: 不直接修改现有 EventBus，而是创建 QueuedEventBus 包装类

**理由**:
- 保持向后兼容，现有代码无需修改
- 允许渐进式迁移，可以在特定场景选择使用队列版本
- 便于回滚，如果发现问题可以直接切换回原版

**替代方案**: 直接修改 EventBus 类
- 拒绝原因：风险高，可能影响所有现有功能

### 2. AgentTaskQueue 使用信号量控制并发

**决策**: `asyncio.Semaphore` + AsyncQueue 组合实现并发控制

**理由**:
- Semaphore 是控制并发数的标准模式
- AsyncQueue 处理任务排队和背压
- 组合使用可以获得两者的优势

**替代方案**: 在队列层面限制（如固定消费者数）
- 拒绝原因：不够灵活，无法动态调整并发数

### 3. WebSocketMessageBuffer 提供 drop/timeout 策略

**决策**: 支持多种满队列策略：阻塞等待、超时丢弃、立即失败

**理由**:
- 不同场景需要不同策略（实时性 vs 可靠性）
- 与 AsyncQueue 已有的 block/timeout 参数保持一致

**替代方案**: 单一策略（始终阻塞）
- 拒绝原因：无法满足低延迟场景需求

### 4. 默认参数保守设置

**决策**:
- EventBus 队列：maxsize=1000
- Agent 并发：max_concurrent=5
- WebSocket 缓冲：maxsize=100

**理由**:
- 保守默认值防止资源耗尽
- 用户可以根据实际需求调整
- 与常见服务器配置相匹配

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 队列引入延迟 | 默认不启用队列模式，按需启用；提供性能监控指标 |
| 队列满导致请求失败 | 提供多种满队列策略；默认值保守；文档明确说明 |
| 内存队列重启丢失 | 当前版本为有意设计；后续可添加持久化后端 |
| 调试复杂度增加 | 添加队列深度、等待时间等监控指标；完善日志 |

## Migration Plan

**部署步骤**:
1. 创建新的队列集成模块（无侵入）
2. 在测试环境验证 QueuedEventBus
3. 逐步在特定对话中启用新功能
4. 监控性能和稳定性

**回滚策略**:
- 纯新增模块，无回滚风险
- 可随时切换回原始 EventBus 实现
- 删除新模块即可完全回滚

## Open Questions

- 是否需要暴露队列指标（深度、等待时间、吞吐量）到监控？
- AgentTaskQueue 是否需要支持优先级（如用户消息优先于后台任务）？
- WebSocket 缓冲是否需要按客户端分离队列？
