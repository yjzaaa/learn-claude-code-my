## Context

项目已经完成了从 TypedDict 到 Pydantic 的模型迁移，但代码库中仍存在大量直接使用裸 dict 的地方。这些裸 JSON 数据主要出现在：

1. 函数参数和返回值类型注解使用 `dict` 或 `Dict[str, Any]`
2. 变量声明直接使用 `{}` 或 `dict()`
3. API 响应直接返回字典而非 Pydantic 模型
4. 事件数据使用裸字典传递

这种实践破坏了类型安全，使得静态分析工具无法有效工作，也增加了运行时错误的风险。

## Goals / Non-Goals

**Goals:**
- 建立代码审计机制，使用正则表达式识别所有裸 JSON 用法
- 为核心模块（core/, managers/, interfaces/）添加完整的类型约束
- 创建测试强制检查，阻止新的裸 JSON 代码进入代码库
- 提供迁移指南和工具，帮助开发者将现有代码转换为类型安全的形式

**Non-Goals:**
- 不修改前端代码
- 不重构业务逻辑，仅添加类型约束
- 不强制要求第三方库的类型注解

## Decisions

### 1. 使用正则表达式进行代码审计

**决策：** 开发一个基于正则表达式的审计工具扫描裸 JSON 模式。

**理由：**
- 正则表达式可以快速识别常见的裸 JSON 模式
- 不需要完整的 AST 解析，执行速度快
- 易于集成到 CI/CD 流程

**识别的模式：**
```python
# 危险的类型注解
Dict\[str,\s*Any\]          # 泛型字典
dict\[str,\s*Any\]          # 小写 dict
dict\s*\(\s*\)              # 空 dict()
\{\s*\}                     # 空 {}
->\s*dict                  # 返回类型为 dict
->\s*Dict\[                # 返回类型为 Dict[...
```

### 2. 测试强制检查机制

**决策：** 创建 `test_no_bare_json.py` 测试文件，使用正则表达式扫描项目文件。

**理由：**
- 测试是阻止代码合并的最有效方式
- pytest 可以很容易地集成到开发工作流
- 失败信息可以明确指出违规文件和行号

### 3. 白名单机制

**决策：** 某些场景允许使用裸 JSON（如与外部 API 交互、配置解析等），提供白名单机制。

**白名单场景：**
- `__init__.py` 文件中的导出
- 与 LangChain 等第三方库的交互点
- 配置文件加载
- 测试文件中的 mock 数据

## Risks / Trade-offs

**[风险] 假阳性过多导致开发者忽视检查** → **缓解措施：** 精细调整正则表达式，提供清晰的错误信息和修复指南

**[风险] 大规模修改引入回归** → **缓解措施：** 分阶段实施，每次只修改一个模块；保持充分的单元测试覆盖

**[风险] 性能影响** → **缓解措施：** Pydantic 模型在序列化时略有开销，但在可接受范围内；使用 `model_validate_json` 等优化方法

## Migration Plan

1. **Phase 1: 审计** - 运行审计工具，生成问题清单
2. **Phase 2: 核心模块** - 优先修改 core/models/ 和 managers/
3. **Phase 3: 接口层** - 更新 interfaces/http/ 和 interfaces/websocket/
4. **Phase 4: 启用检查** - 启用测试强制检查，阻止新的裸 JSON
