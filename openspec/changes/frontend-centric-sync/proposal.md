# 前端中心化消息同步架构重构

## 背景与动机

当前架构中后端同时负责：**Agent Runtime** + **持久化**，这带来以下问题：

1. **刷新页面丢失上下文** - 后端内存中的流式状态无法恢复
2. **服务端负载高** - 需要维护大量活跃会话的完整历史
3. **流式体验脆弱** - 网络中断导致消息丢失，无法恢复
4. **扩展性受限** - 会话状态绑定到具体进程

## 目标架构

分离职责：
- **后端**：Agent Runtime（执行循环 + 临时状态），**不持久化**
- **前端**：消息历史持久化（IndexedDB），状态恢复

## 架构对比

| 维度 | 当前架构 | 目标架构 |
|------|---------|---------|
| **持久化** | 后端内存 + 可选DB | 前端IndexedDB |
| **状态恢复** | 从后端重新加载 | 前端本地恢复 |
| **Agent Loop** | 后端运行 | 后端运行（不变） |
| **流中断恢复** | 不可恢复 | 检查点续传或AI自修复 |
| **离线支持** | 无 | 有限离线（可查看历史） |

## 核心设计原则

1. **职责分离** - Runtime 归后端，Persistence 归前端
2. **乐观优先** - 前端预生成ID，立即更新UI
3. **分层持久化** - 内存 → 批量缓冲 → IndexedDB
4. **KISS** - 接受截断，让AI自修复

## 关键机制

### 1. 消息ID生成
- 前端使用 CUID（时间排序 + 唯一）
- 后端认可并使用该ID

### 2. 流式同步
- 后端发送 `stream:delta`（含chunk序号）
- 前端维护检查点（每N个chunk持久化）
- 中断后可请求 `stream:resume`（从checkpoint继续）

### 3. 状态恢复
- 刷新页面：前端从IndexedDB加载历史
- WebSocket重连：发送 `sync:request` 获取当前流状态
- 后端仅返回活跃流信息，不返回完整历史

### 4. 写入批处理
- 流式内容先写入内存
- 每500ms或50条批量写入IndexedDB
- 页面关闭前 `beforeunload` 强制刷新

## 验收标准

- [ ] 刷新页面后对话历史从IndexedDB恢复
- [ ] 流中断后可恢复或标记截断让AI自修复
- [ ] 快速连续发送消息保持正确顺序
- [ ] 后端重启后前端可重新同步
- [ ] 7天+旧消息自动归档到OPFS

## 排除范围

- 后端不实现消息持久化（纯Runtime）
- 不支持服务端流回放（仅检查点恢复）
- 不实现完整CRDT
- 不支持多设备实时同步

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| IndexedDB满 | 高 | OPFS自动归档旧消息 |
| 前后端ID冲突 | 低 | CUID时间前缀避免冲突 |
| 流恢复失败 | 中 | 降级为截断+AI自修复 |
